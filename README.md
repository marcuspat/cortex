## Overview

Cortex is a personal, always-learning intelligence layer that sits across your entire digital surface area — notes, email, code repositories, bookmarks, chat, calendar — and turns fragmented context into proactive, actionable insight.

The thesis is simple: **people already have a second brain. It just isn't connected.** Notes live in Obsidian. Decisions live in Slack. Work artifacts live in GitHub and Notion. Commitments live in Gmail. No single tool has all the context, so no tool can reason across it. Cortex closes that loop by indexing everything into a unified semantic memory and letting specialized agents operate on that memory continuously.

This repository contains the **web dashboard MVP** — a control plane and observation deck for the Cortex system, built with Next.js 16, TypeScript, and modern UI tooling.

## Features

### Dashboard
- **Real-time system overview** with stats for connectors, memories, insights, and agent health
- **Recent insight cards** highlighting proactive discoveries
- **Agent status grid** showing all 6 specialized agents (Indexer, Researcher, Connector, Drafter, Planner, Orchestrator)
- **Quick actions** for common operations

### Connector Management
- Manage **8 data source integrations**: Gmail, GitHub, Obsidian, Notion, Google Calendar, Google Drive, Slack, Local Filesystem
- Real-time **sync controls** with progress indicators
- Status monitoring: Active, Connecting, Disconnected, Error
- **Add/remove connectors** with guided setup

### Semantic Memory
- Browse your **indexed knowledge base** with full-text search
- Filter by **source type**: Email, Code, Note, Document, Calendar, Chat, Bookmark
- Sort by relevance, date, or access frequency
- **Detail view** with full content, metadata, tags, and source provenance
- Pagination with load-more for large datasets

### Proactive Insights
- **Feed of insight cards** generated by background agents
- Five insight types: Connections, Reminders, Drafts, Summaries, Suggestions
- **Feedback loop** — rate cards as Useful, Not Useful, or Already Knew to tune future surfacing
- Filter by type and status (Pending, Surfaced, Acted, Dismissed)
- Priority scoring (1-10) with visual indicators

### Knowledge Chat
- **Conversational interface** for querying your knowledge base
- Assistant responses with **memory citations** — traceable to source data
- Session management with history
- **Suggested queries** for quick exploration
- Markdown rendering for rich responses

### Agent Monitor
- **Real-time agent status** across all 6 agent types
- **Execution trace browser** with expandable detail views
- Input/output inspection, step-by-step breakdowns
- **Execution timeline** visualization
- Duration tracking and success rate metrics

### Settings
- **Inference configuration**: Provider selection (Claude, OpenAI, Local), model tuning
- **Privacy & security**: Memory retention, auto-decay, audit log, data purge
- **Proactive intelligence**: Quiet hours, rate limits, enable/disable
- **Connector defaults**: Sync interval, auto-sync on startup

```

│  │ Dashboard │ │Connectors│ │  Memory  │ │ Insights │        │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                     │
│  │   Chat   │ │  Agents  │ │ Settings │                     │
│  └──────────┘ └──────────┘ └──────────┘                     │

```

### The Agents

| Agent | Responsibility |
|-------|---------------|
| **Indexer** | Continuously ingests from connected sources. Normalizes, chunks, embeds, and writes to memory. |
| **Researcher** | Answers direct queries by retrieving across memory, ranking, and synthesizing with citations. |
| **Connector** | Runs on a cadence — looks for non-obvious semantic links between recent activity and older memory. Produces insight cards. |
| **Drafter** | Generates draft replies, summaries, and documents when triggers fire. |
| **Planner** | Breaks user goals into actions, sequences them, and surfaces plans for review. |
| **Orchestrator** | Top-level controller. Routes requests, maintains session state, enforces policy. |

### Data Model

```
Connector (1) ──── (N) Memory
    │                    │
    │                    ├── Entity (resolved people, projects, repos)
    │                    │
    └── ConnectorSync

Memory (N) ──── (N) InsightCard (source memories)
Memory (N) ──── (N) ChatMessage (citations)

ChatSession (1) ──── (N) ChatMessage

AgentTrace (execution audit trail)

Setting (key-value configuration)
```

## Getting Started

### Prerequisites

- Node.js (v20+) - [Download](https://nodejs.org/)
- Git
- npm (comes with Node.js)

### Quick Start

```bash
# Clone the repository
git clone https://github.com/marcuspat/cortex.git
cd cortex

# Install dependencies
npm install

# Set up environment variables
cp .env.example .env.local
# Edit .env.local with your configuration

# Push database schema
npm run db:push

# Seed with demo data (optional)
npx tsx prisma/seed.ts

# Start development server
npm run dev
```

The application will be available at `http://localhost:3000`.

### Development Workflow

```bash
# 1. Start development server
npm run dev

# 2. Run tests in watch mode (in another terminal)
npm test -- --watch

# 3. Check linting before committing
npm run lint

# 4. Build for production
npm run build

# 5. Test production build locally
npm run start
```

  5. Run tests (`vitest run`)

View CI status at: `github.com/marcuspat/cortex/actions`

## API Reference

### Dashboard

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/dashboard` | Aggregated stats: connector counts, memory counts, pending insights, recent activity |

### Connectors

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/connectors` | List all connectors with status and sync info |
| `POST` | `/api/connectors` | Create a new connector (`{ type, name }`) |

### Chat

## Design Principles

1. **Local-first, always.** Cloud is a convenience, never a requirement.
2. **Traceable over clever.** Every output must be explainable by pointing at its source memories.
3. **Draft, do not send.** Autonomy is earned, not granted by default.
4. **Depth over breadth.** Ten excellent connectors beat fifty mediocre ones.
5. **The user owns the data**, the schema, and the exit.
6. **One language, one binary.** End-to-end TypeScript; no cross-language IPC.

## Design Patterns

### Domain-Driven Design
The system is organized around 6 bounded contexts:
- **Connector Management** — Data source lifecycle
- **Memory Index** — Semantic knowledge storage
- **Insight Engine** — Proactive surfacing
- **Agent Orchestration** — Multi-agent coordination
- **User Interaction** — Chat and feedback
- **System Configuration** — Settings and policies

See [docs/DDD.md](docs/DDD.md) for the full domain model, and [docs/ARD.md](docs/ARD.md) for the architecture specification.
